# Тестирование возможности записаться на обучение профессии «Тестировщик ПО»

## I.Сценарии для автоматизированных тестов: Тестирование возможности записаться на обучение профессии «Тестировщик ПО»

1. Тестирование через кнопку "Каталог курсов";
2. Тестирование формы подачи заявки через главную страницу;
3. Тестирование формы подачи заявки через страницу "Выберите вектор развития". 

### Позитивное тестирование формы подачи заявки через страницу "Каталог курсов"

**Предусловие: - открытие формы через "Каталог курсов":**

*Шаги*:
1. Открыть сайт https://netology.ru/
2. Нажать на кнопку "Каталог курсов"
3. Выбрать в левой колонке и нажать на кнопку "Программирование"
4. С помощью прокрутки сайта вниз ищем курс Тестировщик ПО и нажимаем на него
**О.Р.** - последовательно открывается страница с каталогом курсов, слева в калонке в первой позиции отображается "Програмирование", после нажатия внизу  отображается курс "Тестировщик ПО", после нажатия на него мышкой появляется страница с информацией по курсу

**Тестирование формы отправки:**
*Шаги*:
1. Нажимаем на кнопку "записаться"
2. В появивишемся окне вводим валидные значения имени и телефона
3. Нажимаем на кнопку "записаться"

**О.Р. - форма заполняется ввалидными значениями и запрос уходит в БД, появляется окно "мы скоро с вами свяжемся"**

*Тестовые данные:*
1. Имя - валидные значения имени на русском/английском языке (Валентина, Valentina);
2. телефон - валидное значение в виде 11 цифр формата ( +79090009900);

## II. Перечень используемых инструментов:
1. Использование базы данных из Докера - для моделирования отправки реальных данных по форме и анализа ее появления
2. Написание автотестов в программе "Идея"
3. Язык програмирование - Java (11 версия)
4. Для поиска локаторов и тестирования кнопок и форм для ввода данных используется Devtools;
5. Поиск локаторов и методов воздействия на них через методы Selenid 
6. Использование Design Patterns - для создания шаблонов и сокращения копипаста в коде
7. Генератор данных (напимер Faker) для создания тестовых данных
8. Page Object - создание отдельных страниц кода вне теста для страницы с информацией по курсу >Тестировщик ПО>, для формы ввода данных - для уменьшения кода в тестах и также облегчения корректировки в случае, если локаторы поменяются

## III. Перечень необходимых разрешений, данных и доступов:
1. Разрешение владельца сайта на доступ к тестовой странице
2. Доступ к базе данных сайта
4. Доступ к API сайта
3. Доступ к основному ТЗ с информацией,а  что должно быть при позитивных сценариях
4. Статистика, в которой отображается, какой путь для пользователей самый основной, чтобы проверить его в первую очередь

## IV. Перечень и описание возможных рисков при автоматизации:
1. Некорректная интеграция с БД - основной функционал по факту не будет проверятся
2. отсутсвие четких локаторов для элементов, необходимых для авто-теста
3. Частые изменения локаторов элементов при обновлении сайта
4. Слишком большое количество ненужных авто-тестов, особенно негативных
5. При отсутствии статистики по самому предпочтительному пути поиска и заполнения формы риск не проверить ее в первую очередь
6. При отсутсвии четкого ТЗ риск пропустить возможность ввода невалидных значений (например - на английском языке нельзя, но форма пропускает, в ТЗ этого тоже нет, в итоге невозможность обработки данных на этапе БД)
